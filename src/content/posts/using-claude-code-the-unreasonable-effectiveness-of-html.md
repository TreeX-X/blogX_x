# 为什么使用 HTML？

有几项因素使 HTML 成为比我目前在 Claude Code 中使用 Markdown 更适合的选择，特别是对于那些需要或包含以下任务的工作：

## 信息密度（Information density）

HTML 相比 Markdown 可以传达更丰富的信息。当然，它可以像 Markdown 一样做简单的文档结构，如标题和格式化，但它还可以表示各种其他信息，例如：

使用表格表示表格数据

使用 CSS 表示设计数据

使用 SVG 表示插图

使用 script 标签表示代码片段

使用带有 javascript + CSS 的 HTML 元素表示交互

使用 SVG 和 HTML 表示工作流

使用绝对位置和画布表示空间数据

使用 image 标签表示图像

在我看来，几乎没有 Claude 可以读取的信息集，你不能用 HTML 有效表示。这使得它成为模型向你传达深度信息以及你审阅信息的高度有效方式。

我发现，在没有这种能力的情况下，模型可能会在 Markdown 中做更多低效的事情，比如 ASCII 图表，或者我最喜欢的，用 Unicode 字符估算颜色。

## 视觉清晰度和易读性（Visual clarity and ease of reading）

随着 Claude 能够处理更复杂的工作，它也能够编写越来越大的规范和计划。我发现我实际上往往不会阅读超过 100 行的 Markdown 文件，而且我肯定无法让组织中的其他人去阅读它。

但 HTML 文档更容易阅读，因为 Claude 可以直观地组织结构，使其适合使用标签、插图和链接进行导航。它甚至可以是移动响应式的，因此你可以根据不同的设备形式因素以不同方式阅读它。

## 易于共享（Ease of sharing）

Markdown 文件相当难以共享，因为大多数浏览器不能很好地原生渲染它们。你通常必须将它们作为附件添加到电子邮件或消息中。

只要你上传 HTML 文件，就可以轻松共享链接。你的同事可以在他们希望的任何地方打开它并轻松引用它。

如果你的规范、报告或 PR 说明以 HTML 格式呈现，有人实际阅读它的机会会大得多。

## 双向交互（Two-way interactions）

HTML 还允许你与文档进行交互；例如，你可能想要它添加滑块或旋钮来调整设计，或者允许你调整算法中的不同选项以查看会发生什么。你也可以要求它让你将这些更改复制到提示中，然后粘贴回 Claude Code。

在有用的情况下，这可以让你为你正在处理的特定问题创建单独的编辑环境。

## 数据摄取（Data ingestion）

使用 Claude Code 而不是 Claude.ai 或 Claude Design 来制作 HTML 文件的最大原因之一是 Claude Code 可以摄取大量上下文信息。例如，在撰写本文时，我要求 Claude Code 阅读我的代码文件夹，找到我生成的所有 HTML 文件，对它们进行分组和分类，然后制作一个包含代表每种类型的图表的 HTML 文件。您在本文中看到的图表就是这一过程的直接结果。

除了文件系统，Claude Code 还可以通过您的 MCP（如 Slack、Linear 等）、您的网络浏览器（在 Chrome 中使用 Claude）以及您的 git 历史记录来找到额外的上下文信息。

入门指南

有一点值得注意：要让 Claude 生成这样的 HTML 文件，您不需要做太多工作。您可以简单地提示它"制作一个 HTML 文件"或"制作一个 HTML 工件"。关键是您希望这个工件做什么以及如何使用它。随着时间的推移，围绕重复模式构建技能可能是有意义的，但从头开始提示是了解它在不同用例中如何运作的好方法。

用例

为了使这种方法更加具体，以下是一些示例用例，我认为在这些情况下使用 HTML 文件比 Markdown 更有意义。您也可以通过这些用例的 GitHub 图库跟进，[这里](https://github.com/anthropics/claude-code-examples)。

规范、规划和探索

HTML 是 Claude 深入研究问题的丰富画布。当我开始处理一个问题时，我期望创建一个 HTML 文件网络，而不是简单的 Markdown 计划。例如，我可能会先要求 Claude Code 头脑风暴并创建一些不同选项的探索方案。然后我会要求它深入其中某一个，可能制作该类型界面的模型或示例。最后，当我感觉良好时，我会要求它编写一个实施计划。当我满意计划后，我会创建一个新会话并传入所有这些文件供其实现。

在验证时，我也会要求验证代理读取这些文件，这样它就能对所需内容有更广泛的了解。

示例提示：

我不确定入职屏幕应该采用什么方向。生成 6 种明显不同的方法——改变布局、语调和密度——并将它们以网格形式布局在单个 HTML 文件中，以便我并排比较。为每个方法标注其做出的权衡。

在 HTML 文件中创建一个详细的实施计划，确保制作一些模型，展示数据流，并添加我可能需要查看的重要代码片段。使其易于阅读和理解。

用途：

探索在代码中实现其他方法

同时尝试多种视觉设计

代码审查和理解

代码在 Markdown 文件中可能难以阅读，但使用 HTML，我们可以渲染差异、注释、流程图和模块。使用 HTML 来理解代理编写的代码、审查代码，或向审查您代码的人解释 PR。

示例提示：

通过创建描述 PR 的 HTML 工具来帮助我审查这个 PR。我对流控/背压逻辑不太熟悉，所以请重点关注这部分。渲染实际的差异并带有内边距注释，按严重程度对发现进行颜色编码，以及任何其他可能有助于清晰传达概念的内容。

用于：

创建 PR

审查 PR

理解代码中的主题

设计和原型

Claude Design 基于 HTML，因为 HTML 在设计方面具有极强的表现力，即使你的最终界面不是 HTML。Claude 可以用 HTML 勾勒出设计，然后用你选择的语言（如 React、Swift 等）来实现它。

你还可以制作交互原型，例如动画、操作等。可以考虑要求 Claude 创建滑块、旋钮等，以精确调整你所需的内容。

‍

示例提示：

我想原型化一个新的结账按钮，点击时它会有一个播放动画，然后迅速变成紫色。请为我创建一个 HTML 文件，其中包含几个滑块和选项，让我可以尝试不同的动画效果，给我一个复制按钮，用于复制效果良好的参数。

用于：

创建设计系统工具

调整组件

可视化组件库

原型化动画

报告、研究和学习

Claude Code 在综合多个数据源的信息并将其转换为可读报告方面非常有效。你可以提示 Claude 搜索你的 Slack、代码库、git 历史记录或互联网，并使用它生成易于阅读的报告。

你可以将其组装成一个长 HTML 文档、交互式解释器，甚至是幻灯片/演示文稿。要求 Claude 使用 SVG 制作图表以帮助可视化。

示例提示：

我不理解我们的速率限制器实际上是如何工作的。阅读相关代码并生成一个单一的 HTML 解释页面：一个令牌桶流程的图表，3-4个关键代码片段的注释，以及底部的"陷阱"部分。优化它以便一次性阅读。

用于：

编写功能摘要

生成解释器

起草每周状态报告

创建事件报告

生成 SVG 插图、流程图和技术图

自定义编辑界面

有时很难在文本框中纯粹描述你想要的东西。对于这个用例，我经常要求 Claude 为我正在处理的特定内容构建一个一次性编辑器：不是产品或可重用工具，而是一个单一的 HTML 文件，专门为这一块数据而设计。

技巧总是以导出结束：一个"复制为 JSON"或"复制为提示"按钮，将我在 UI 中所做的任何操作转换回可以粘贴到 Claude Code 或提交到文件的内容中。你仍然在循环中，但循环变得更紧密。

示例提示：

我需要重新排序这30个Linear工单。请创建一个HTML文件，将每个工单作为可拖拽卡片分布在"现在/下一步/稍后/删除"列中。请根据您的最佳猜测进行预排序。添加一个"复制为Markdown"按钮，用于导出最终排序结果，并为每个桶提供一行理由说明。

这是我们的功能标志配置。请为其构建一个基于表单的编辑器，按区域分组标志，显示它们之间的依赖关系，如果启用了一个其先决条件被关闭的标志，请警告我。添加一个"复制差异"按钮，只提供已更改的键值。

我正在调整这个系统提示。请创建一个并排编辑器：左侧是可编辑的提示，其中变量插槽高亮显示；右侧是三个示例输入，它们会实时重新渲染填充后的模板。添加一个字符/标记计数器和复制按钮。

用途包括：

重新排序、分类或分组任何内容（工单、测试用例、反馈）

编辑结构化配置（功能标志、环境变量、带约束的JSON/YAML）

调整提示、模板或文案，并实时预览

策划数据集 — 批准/拒绝行，标记示例，导出所选内容

注释文档、转录内容或差异，并导出注释

选择难以用文本表达的值：颜色、缓动曲线、裁剪区域、cron表达式、正则表达式

常见问题

以下是我关于使用HTML与Claude Code时被问得最多的问题，以及我形成的日常实用习惯：

这不是效率更低吗？

虽然Markdown通常使用的标记更少，但我发现HTML的表达能力更强，而且我阅读它的可能性也高得多，因此我能获得更好的整体输出。在Opus 4.7的100万上下文窗口中，增加的标记使用量在上下文窗口中并不明显。

现在什么时候使用Markdown？

老实说，我已经几乎完全停止使用Markdown来做几乎所有事情，但我可能已经是HTML极端主义者的那一端了。

这是你替代规划的方式吗？

我发现，与其制定单一计划，我倾向于为计划的不同部分/阶段准备几个不同的HTML文件。例如，我可能会用HTML制定一个实施计划，然后做一个用于探索UI的文件，最后再创建一个列出每个设计的HTML组件。我倾向于保留这些文件作为未来的参考，也用于验证。

与Claude保持同步

以上所有内容都是为了说明，我使用HTML而不是Markdown的真正原因是它帮助我更紧密地与Claude保持同步。随着Claude承担更多工作，我注意到自己不太仔细地阅读计划，我想要一种方法来参与它的选择，而不是简单地交接出去。HTML恰好就是那个方法。我现在感觉比以往任何时候都更紧密地参与其中。
